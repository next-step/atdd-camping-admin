# 최우선 기능 상세 분석 및 인수 테스트 시나리오

`api_risk_assessment.md`에서 도출된 리스크 점수 8점의 최우선 기능에 대한 **5W1H 분석**, **핵심 인수 조건**, **검증 포인트**를 정리했습니다.

---

## 대여 생성 (`POST /admin/rentals`)

### 5W1H 분석
*   **Who (누가)**: 캠핑장 관리자 (Admin)
*   **When (언제)**: 고객이 현장에서 장비를 추가로 빌리거나, 예약 시 신청한 장비를 수령할 때
*   **Where (어디서)**: 관리자 대시보드의 대여 관리 페이지에서
*   **What (무엇을)**: 특정 대여용 상품(Product)을 지정된 수량만큼 대여 처리하고 기록을 남긴다.
*   **Why (왜)**: 추가 매출을 발생시키고, 자산(장비)의 반납 여부를 관리하며, 실시간 재고를 정확하게 유지하기 위함이다.
*   **How (어떻게)**: 상품 ID, 대여 수량, (선택) 예약 ID를 입력받아 시스템에 등록하면, 즉시 해당 상품의 재고를 차감하고 대여 레코드를 생성한다.

### 핵심 인수 조건 (Key Acceptance Criteria)
> **"관리자는 '대여 가능(RENTAL)' 타입의 상품을 '현재 재고 내'에서 요청한 수량만큼 대여 처리할 수 있으며, 성공 시 즉각적인 재고 차감이 이루어져야 한다."**

### 핵심 검증 포인트 (Verification Points)
1.  **상품 유효성 검증**:
    *   존재하지 않는 상품 ID 요청 시 적절한 에러(4xx)가 발생하는가?
    *   상품 타입이 `RENTAL`이 아닌 경우(예: `SALE`) 대여가 차단되는가?
2.  **재고 검증 (Critical)**:
    *   요청 수량이 현재 재고량보다 즐 때(`Quantity > Stock`) 대여가 실패하는가?
    *   대여 성공 후, 상품의 재고(`StockQuantity`)가 요청 수량만큼 정확히 줄어들었는가?
3.  **동시성 제어 (Concurrency)**:
    *   재고가 1개 남았을 때 동시에 2명이 대여 요청을 하면 1명만 성공하고 1명은 실패하는가? (Race Condition 방지)
4.  **예약 연동**:
    *   유효한 `reservationId`가 주어졌을 때, 생성된 대여 기록이 해당 예약과 올바르게 연결되는가?
