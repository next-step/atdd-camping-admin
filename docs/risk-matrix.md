# 리스크 매트릭스

## 평가 기준

### 척도 정의
- **높음(3점)**: 심각한 영향 / 빈번함 / 높은 발생 확률
- **보통(2점)**: 중간 수준의 영향 / 간헐적 / 중간 발생 확률
- **낮음(1점)**: 경미한 영향 / 드묾 / 낮은 발생 확률

### 계산식
```
리스크 점수 = 비즈니스 임팩트 + 변경 빈도 + 버그 가능성
```

### 평가 항목 설명
| 항목 | 설명 |
|------|------|
| 비즈니스 임팩트 | 기능 장애 시 매출, 운영, 고객 경험에 미치는 영향 |
| 변경 빈도 | 해당 기능이 수정/확장될 가능성 및 코드 복잡도 |
| 버그 가능성 | 현재 코드에서 발견된 잠재적 결함 수준 |

## Top 3 선정 근거

### 1위. 대여 생성 (9점) - `POST /admin/rentals`

| 평가 항목 | 점수 | 근거 |
|----------|------|------|
| 비즈니스 임팩트 | 높음(3) | 매출과 직결, 재고 관리 오류 시 실물 자산 손실 |
| 변경 빈도 | 높음(3) | 재고-예약-대여 간 복잡한 연관관계, 확장 가능성 높음 |
| 버그 가능성 | 높음(3) | **동시성 이슈** (Race Condition으로 재고 음수 가능), **음수 quantity 취약점** (재고 조작 가능) |

**발견된 주요 결함**:
```java
// ProductService.java - 동시성 미제어
if (product.getStockQuantity() < quantity) { // Read
    throw new IllegalStateException(...);
}
product.setStockQuantity(product.getStockQuantity() - quantity); // Write
// 두 스레드가 동시 접근 시 Race Condition 발생
```

```java
// quantity 검증 없음 - 음수 입력 시 재고 증가
RentalRecord rentalRecord = new RentalRecord(reservation, product, quantity);
// quantity = -10 → 재고가 오히려 증가
```

**권장 테스트 케이스**:
- [ ] 동시 대여 요청 시 재고 정합성 테스트
- [ ] quantity 음수/0 입력 시 거부 테스트
- [ ] SALE 타입 상품 대여 거부 테스트
- [ ] 재고 부족 시 예외 처리 테스트

### 2위. 예약 상태 변경 (8점) - `PATCH /admin/reservations/{id}/status`

| 평가 항목 | 점수 | 근거 |
|----------|------|------|
| 비즈니스 임팩트 | 높음(3) | 예약 확정/취소는 매출 및 고객 경험의 핵심 |
| 변경 빈도 | 보통(2) | 상태 전이 로직 추가 시 변경 필요 |
| 버그 가능성 | 높음(3) | **상태 검증 누락** (임의 문자열 저장 가능), **상태 전이 규칙 없음** (CANCELLED→CONFIRMED 가능) |

**발견된 주요 결함**:
```java
// ReservationAdminController.java - enum 검증 없음
String statusValue = statusObj.toString();
reservation.setStatus(statusValue); // "INVALID_STATUS" 저장 가능

// ReservationStatus enum이 정의되어 있으나 사용하지 않음
public enum ReservationStatus {
    WAITING, PENDING, CONFIRMED, REJECTED, CHECKED_IN, CHECKED_OUT, CANCELLED
}
```

**권장 테스트 케이스**:
- [ ] 유효하지 않은 상태 값 거부 테스트
- [ ] 허용되지 않는 상태 전이 거부 테스트 (예: CANCELLED → CONFIRMED)
- [ ] null/빈 문자열 상태 값 처리 테스트
- [ ] 존재하지 않는 예약 ID 처리 테스트

### 3위. 상품 생성 (6점) - `POST /admin/products`

| 평가 항목 | 점수 | 근거 |
|----------|------|------|
| 비즈니스 임팩트 | 보통(2) | 잘못된 상품 데이터는 대여/판매에 영향 |
| 변경 빈도 | 보통(2) | Map 파싱 로직 복잡, DTO 리팩토링 예상 |
| 버그 가능성 | 보통(2) | **입력 검증 부재** (null name, 음수 price 허용), **조용한 실패** (잘못된 입력 시 기본값 적용) |

**발견된 주요 결함**:
```java
// ProductAdminController.java - 조용한 실패
try {
    stockQuantity = Integer.valueOf(v.toString());
} catch (Exception e) {
    stockQuantity = 0; // 오류 숨김
}

// name이 null이면 DB 저장 시 예외 발생
Product newProduct = new Product(name, stockQuantity, price, productType);
// @Column(nullable = false) 위반 → DataIntegrityViolationException
```

**권장 테스트 케이스**:
- [ ] name null/빈 문자열 거부 테스트
- [ ] price 음수 거부 테스트
- [ ] stockQuantity 음수 거부 테스트
- [ ] 잘못된 productType 처리 테스트

## 리스크 완화 권장사항

│  1. 재고 관리에 비관적 락(@Lock) 또는 낙관적 락(@Version) 적용  │
│  2. quantity 양수 검증 추가 (@Min(1))                        │
