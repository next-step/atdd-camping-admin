# 시스템 분석 결과

## 개요
- Spring Boot 기반 단일 서비스로, 예약·캠핑 구역·상품·대여·매출을 하나의 관리자 화면에서 다룬다.
- 화면용 컨트롤러와 관리자용 데이터 제공 컨트롤러가 분리되어 있어, 동일한 도메인 로직을 여러 채널에서 재사용한다.
- 모든 데이터는 H2 메모리형 데이터베이스에 저장되며, 기동 시 준비된 자료(data.sql)가 자동으로 채워져 시연과 학습에 활용된다.
- 로그인 검사를 거치지 않으면 어떤 관리 화면도 볼 수 없고, 인증 필터가 요청마다 토큰을 확인한다.

## 핵심 구성 요소
- **도메인 모델**: 예약, 캠핑 구역, 상품, 대여 기록, 매출 기록, 고객 정보가 각각 독립된 엔터티로 구성된다.
- **서비스 계층**: 상품 재고 관리, 대여·판매 처리, 매출 산출처럼 업무 규칙이 복잡한 부분을 담당한다.
- **데이터 창고**: Spring Data JPA 저장소가 각 엔터티의 데이터 입출력을 담당하며, 기본 CRUD 외에도 중복 예약 탐색, 미반납 대여 조회 같은 맞춤 쿼리를 제공한다.
- **보안 모듈**: JWT 발급기와 인증 필터가 로그인 성공 시 발급한 토큰을 검사하여 관리자만 기능을 사용할 수 있도록 보호한다.
- **웹 레이어**: 템플릿 기반 화면 컨트롤러가 예약·캠핑 구역·상품·매출 화면을 렌더링하며, 관리 API는 JSON 응답으로 데이터를 제공한다.

## 비즈니스 규칙 (유비쿼터스 언어)

- **관리자 전용**: 로그인에 성공한 관리자만 모든 기능을 사용할 수 있다.
- **예약 기본값**: 예약은 저장되는 순간 자동으로 “확정” 상태가 되고, 기록 시각이 설정된다.
- **필수 예약 정보**: 예약자 이름, 이용 시작일, 이용 마감일, 선택한 캠핑 구역이 비어 있으면 예약을 저장할 수 없다.
- **중복 검증 부재**: 동일 구역·기간 중복을 자동으로 가려내지 않으므로 담당자가 수동으로 일정 충돌을 확인해야 한다.
- **확인 코드 처리**: 확인 코드와 전화번호는 선택 입력이며, 시스템이 자동으로 값을 채워 주지 않는다.
- **캠핑 구역 고유성**: 캠핑 구역 번호는 서로 달라야 하며 중복되면 저장이 거절된다.
- **재고 보호**: 판매나 대여 시 재고가 부족하면 작업이 중단되고 재고가 음수로 내려가지 않는다.
- **대여 반납**: 대여 품목을 반납 처리하면 곧바로 재고가 복구되고, 이미 반납된 기록은 두 번 반납할 수 없다.
- **매출 계산 기준**: 숙박 매출 합산 시 1박당 5만 원을 기준 단가로 사용한다.
- **초기 데이터 재생성**: 서비스가 다시 시작되면 준비된 샘플 데이터가 매번 새로 채워져 동일한 초기 상태가 유지된다.

- **인증 및 접근 제어**
  - 관리자 인증에 통과한 세션만 모든 관리 API와 화면에 접근할 수 있으며, JWT 토큰이 만료되면 즉시 재로그인이 필요하다.
  - 로그인 응답은 본문 토큰과 HTTP 전용 쿠키 두 경로로 전달되어, 클라이언트는 어느 방식으로든 인증을 이어갈 수 있다.

- **예약 도메인 규칙**
  - 예약 엔터티는 `customerName`, `startDate`, `endDate`, `campsite` 필드가 비어 있을 경우 영속화되지 않는다.
  - `@PrePersist` 훅이 동작해 새 예약의 `status`를 기본값 `CONFIRMED`로, `createdAt`을 현재 시각으로 설정한다.
  - 시스템에는 동일 구역·기간 중복을 탐지하는 쿼리가 존재하지만 컨트롤러 레벨에서 호출되지 않아, 운영자가 중복 접수를 수동 검증해야 한다.
  - 확인 코드(`confirmationCode`)와 전화번호(`phoneNumber`)는 nullable로 설계되어 있어, 필요 시에만 수동 입력한다.

- **캠핑 구역 관리 규칙**
  - 캠핑 구역 번호(`siteNumber`)는 Unique 제약이 걸려 있어 중복 저장 시 예외가 발생한다.
  - 번호 패턴으로 구역 크기나 타입을 유추하는 로직은 없으며, 설명과 최대 인원 필드는 운영 가이드용 메타데이터로만 사용된다.

- **상품·재고 도메인 규칙**
  - 상품 재고는 음수가 될 수 없으며, 판매·대여 서비스 층에서 수량이 부족하면 `IllegalStateException`을 발생시켜 트랜잭션을 롤백한다.
  - 상품 유형은 `SALE`과 `RENTAL` 두 값만 허용되며, 대여 생성 시 유형 검증을 통과해야 한다.

- **대여 프로세스 규칙**
  - 대여 등록 시 예약 연결은 선택 사항이며, 예약 ID가 없으면 독립 워크인 대여로 저장된다.
  - 반납 처리 시 `isReturned` 플래그로 중복 반납을 차단하고, 재고 보충은 동일 트랜잭션 내에서 보장된다.

- **매출 및 보고 규칙**
  - 숙박 매출 산정은 숙박 박수 × 50,000원 고정 단가로 계산하며, 1박 미만은 최소 1박으로 처리한다.
  - 기간 합산 보고는 예약, 판매, 대여 기록을 모두 메모리에서 필터링해 계산하므로 데이터 규모 증가 시 퍼포먼스 이슈가 발생할 수 있다.

- **비기능·운영 규칙**
  - 애플리케이션 부팅 시 `data.sql`이 실행되어 동일한 시드 데이터가 매번 주입된다.
  - 로그 레벨은 도메인 패키지와 SQL에 대해 DEBUG 이상으로 설정되어 있어, 운영 환경에서는 로그 노이즈가 다소 높을 수 있다.

## 운영상 고려 사항
- 예약 등록·연박 검증·환불 계산이 수동 절차에 의존하므로, 업무 매뉴얼 정비와 추가 자동화가 필요하다.
- 매출 단가가 코드에 고정되어 있어 가격 정책이 바뀔 때마다 개발 작업이 요구된다.
- 관리자 인증 토큰이 30분 후 만료되므로, 장시간 작업 시 재로그인이 필요함을 운영자에게 안내해야 한다.
- 로그가 매우 상세하게 기록되므로, 문제 발생 시 로그 분석으로 원인을 빠르게 추적할 수 있다.
