## Top Risk 상세 분석

### 1. 예약 생성 및 검증 부재 (총점 8)
- **위험 요인**: 현재 API와 UI에 신규 예약 저장 로직이 구현되어 있지 않아, 운영자가 DB 조작이나 외부 시스템을 통해 예약을 만들어야 한다. 생성 시 필수 검증(과거 날짜 차단, 30일 제한, 중복 여부, 확인 코드 생성)이 모두 수동 절차에 의존한다.
- **잠재 영향**: 잘못된 일정이나 중복 예약이 그대로 시스템에 반영되어 현장 이중 배정, 고객 불만, 환불 분쟁으로 이어질 수 있다.
- **기술 메모**: `ReservationRepository.findOverlappingReservations`는 존재하지만 서비스에서 호출되지 않는다. `@PrePersist`로 기본 상태만 설정되므로 전체 생성 플로우가 필요하다.
- **권장 대응**:
  1. 예약 생성 REST API와 관리자 화면을 구현하고, 프론트/백엔드 모두에서 기간·필수 필드 검증을 추가한다.
  2. 서비스 계층에서 중복 쿼리를 호출해 원자적으로 겹치는 예약을 차단하도록 트랜잭션 격리를 높이거나 `SELECT FOR UPDATE`를 고려한다.
  3. 확인 코드 생성기를 도입해 난수 기반 6자리 코드를 자동 발급하고, 상태 변경 시 본인 인증에 활용한다.

### 2. 가용성 판단(연박 포함) (총점 8)
- **위험 요인**: 특정 날짜 또는 기간에 비어 있는 캠핑 구역을 계산하는 API가 없다. 연박 가능 여부도 시작일·마감일만 비교하여 판단하므로, 중간 날짜를 검증하지 못한다.
- **잠재 영향**: 예약 상담 시 잘못된 안내가 이뤄져 실제 현장 가용 구역이 없는데도 예약을 받거나, 반대로 가능함에도 거절하여 매출 손실이 발생할 수 있다.
- **기술 메모**: 중복 검사를 위한 JPQL 쿼리가 존재하지만 호출되지 않으며, Batch/Stream 필터링으로 해결하려면 성능 문제가 발생한다.
- **권장 대응**:
  1. `ReservationRepository.findOverlappingReservations`를 서비스에서 사용하여 범위 겹침을 판별한다.
  2. 기간 검색 API를 추가해 연속 며칠을 모두 점검하고, 필터 옵션(구역 유형, 수용 인원)을 제공한다.
  3. 예약 캘린더 뷰에 동일 로직을 적용해 월별로 빈자리와 예약 상태를 시각적으로 표시한다.

### 3. 매출 보고 및 단가 정책 (총점 8)
- **위험 요인**: 숙박 매출 합산이 1박당 50,000원으로 하드코딩 되어 있어 정책이 바뀌면 즉시 코드 수정을 요구한다. 또한 보고 로직이 전체 데이터를 메모리로 가져와 필터링하므로 데이터가 늘어나면 성능 저하 위험이 있다.
- **잠재 영향**: 단가 변경을 반영하지 못하면 재무 보고가 왜곡되어 경영 판단을 잘못 내릴 수 있고, 성수기에는 보고서 생성 시간이 급증한다.
- **기술 메모**: `SalesService.generateDailyRevenueReport` 등에서 각각의 데이터를 Stream으로 필터링 후 합산한다. 예약 박수 계산 시 최소 1박 로직이 있으나 테스트 커버리지가 부족하다.
- **권장 대응**:
  1. 숙박 단가를 설정 파일 또는 구역별 요금 테이블로 외부화하고, 정책 변경 시 운영자가 값만 조정하도록 한다.
  2. 통계 계산을 SQL 수준(aggregation query)으로 이관하거나, 예약/판매/대여 테이블에 요약 데이터를 캐싱하는 배치 작업을 도입한다.
  3. 장기 보관용 보고서는 별도 데이터 웨어하우스로 분리해 운영 DB 부하를 줄인다.

## 기타 주의 영역
- **재고·대여 동시 처리**: 재고 보호 로직은 존재하지만, 동시 다발 대여/판매 시 ProductService가 단순 뺄셈 방식이라 Race Condition 위험이 있다. DB 레벨 Lock 또는 Version Column 도입이 필요하다.
- **관리자 인증**: 토큰 만료가 30분이어서 장시간 근무 시 반복 로그인이 필요하다. SSO나 Refresh Token 없이 쿠키에만 의존하는 구조라 세션 탈취에 민감하다.
- **로그 볼륨**: DEBUG/TRACE 수준 로그가 운영 환경에도 적용되어 있어, 장기간 운영 시 로그 저장소 비용과 분석 속도에 부담을 준다.

## 개선 우선순위 제안
1. 예약 생성/가용성 판단의 자동화와 동시성 제어 보강
2. 매출 보고의 단가 설정 분리 및 계산 성능 개선
3. 재고/대여 트랜잭션에 대한 낙관/비관 잠금 전략 수립
4. 인증 세션 관리 강화 및 운영 로그 정책 정비
