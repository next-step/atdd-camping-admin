## 고위험 API 상세 분석

### 1. 예약 상태 변경 (총점: 14)
```
PATCH /admin/reservations/{id}/status
```

| 리스크 요소 | 설명 |
|------------|------|
| 비즈니스 임팩트 (5) | 잘못된 상태 변경 → 고객 불만, 매출 손실, 노쇼 처리 오류 |
| 변경 빈도 (4) | 상태 전이 규칙 추가, 알림 연동, 환불 정책 변경 가능 |
| 버그 가능성 (5) | 상태 전이 검증 없음, Enum 유효성 검사 없음, 동시성 이슈 |

**주요 테스트 시나리오:**
- CONFIRMED → CANCELLED 상태 변경
- 유효하지 않은 상태값 입력
- 존재하지 않는 예약 ID
- 이미 CANCELLED된 예약 재취소

---

### 2. 대여 생성 (총점: 12)
```
POST /admin/rentals
```

| 리스크 요소 | 설명 |
|------------|------|
| 비즈니스 임팩트 (5) | 재고 불일치 → 초과 대여, 매출 손실 |
| 변경 빈도 (3) | 대여 기간 추가, 보증금 처리 등 확장 가능 |
| 버그 가능성 (4) | 재고 검증, RENTAL 타입 검증, nullable reservationId |

**주요 테스트 시나리오:**
- 정상 대여 생성 (예약 연결)
- Walk-in 대여 (reservationId 없음)
- 재고 부족 시 대여 실패
- SALE 타입 상품 대여 시도

---

### 3. 판매 처리 (총점: 12)
```
POST /api/sales
```

| 리스크 요소 | 설명 |
|------------|------|
| 비즈니스 임팩트 (5) | 매출 직결, 재고 관리 핵심 |
| 변경 빈도 (3) | 할인 적용, 결제 연동 확장 가능 |
| 버그 가능성 (4) | 다중 아이템 처리, 재고 검증, 트랜잭션 처리 |

**주요 테스트 시나리오:**
- 단일/다중 상품 판매
- 재고 부족 시 판매 실패
- 존재하지 않는 상품 ID

---

### 4. 대여 반납 (총점: 11)
```
PATCH /admin/rentals/{id}/return
```

| 리스크 요소 | 설명 |
|------------|------|
| 비즈니스 임팩트 (5) | 재고 복구 실패 → 재고 불일치 |
| 변경 빈도 (2) | 비교적 안정적 |
| 버그 가능성 (4) | 중복 반납 처리, 재고 동기화 |

**주요 테스트 시나리오:**
- 정상 반납 및 재고 복구 확인
- 이미 반납된 대여 재반납 시도
- 존재하지 않는 대여 ID

---

### 5-6. 수익 보고서 (총점: 11)
```
GET /admin/reports/revenue/daily
GET /admin/reports/revenue/range
```

| 리스크 요소 | 설명 |
|------------|------|
| 비즈니스 임팩트 (4) | 경영 의사결정 영향, 정산 오류 |
| 변경 빈도 (3) | 보고서 형식, 필터 조건 변경 가능 |
| 버그 가능성 (4) | 3개 도메인 통합 계산, 날짜 범위 처리 |

**주요 테스트 시나리오:**
- 예약/판매/대여 수익 정확성
- 빈 데이터 기간 조회
- 날짜 범위 경계값

## 테스트 우선순위 권장

### 1순위 (필수)
| API | 이유 |
|-----|------|
| 예약 상태 변경 | 가장 높은 리스크, 상태 관리 핵심 |
| 대여 생성 | 재고 관리 핵심, 복잡한 검증 로직 |
| 판매 처리 | 매출 직결, 공개 API |

### 2순위 (권장)
| API | 이유 |
|-----|------|
| 대여 반납 | 재고 복구 정확성 |
| 수익 보고서 | 복잡한 계산 로직 |
| 로그인 | 보안 관련 |

### 3순위 (선택)
| API | 이유 |
|-----|------|
| 상품/캠프장 CRUD | 단순 CRUD, 낮은 복잡도 |
| 목록 조회 API | 단순 조회 |
