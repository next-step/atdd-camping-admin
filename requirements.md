# 초록 캠핑장 관리자 기능 명세서

## 1. 예약 관리 기능

### 1.1 예약 등록 준비
- **예약 입력 방식**: 새 예약을 직접 입력하는 절차가 아직 준비되지 않아, 담당자가 별도 도구나 DB 변경을 통해 추가해야 한다.
- **필수 기초 정보**: 예약자 이름, 이용 시작일, 이용 마감일, 선택한 캠핑 구역이 모두 채워져 있어야 저장된다.
- **자동 채움 항목**: 예약이 저장되면 작성 시각과 기본 상태(확정)가 자동으로 기록된다.
- **비어 있어도 되는 항목**: 전화번호와 확인 코드는 비워 둔 채로도 저장된다.
- **확인 코드 생성**: 여섯 자리 확인 코드를 자동으로 만들어 주지 않아 담당자가 직접 입력하거나 빈칸으로 둔다.

### 1.2 일정 검증과 연박 처리
- **날짜 검증 책임**: 과거 날짜, 30일 이후 일정, 시작일보다 빠른 마감일 등을 시스템이 막아 주지 않으므로 담당자가 직접 확인해야 한다.
- **연박 확인**: 시작일과 마감일만 저장되며, 사이 날짜가 다른 예약과 겹치는지 자동으로 살피지 않는다.

### 1.3 예약 조회와 검색
- **예약 목록**: 관리자는 예약 현황 표에서 모든 예약을 한 번에 확인할 수 있다.
- **검색 기능 부재**: 특정 고객, 날짜, 전화번호로 좁혀 보는 기능이 없어, 필요한 경우 목록을 내려받아 수동 필터링해야 한다.
- **개별 상세 보기**: 예약 한 건만 따로 보여주는 화면은 아직 마련되지 않았다.

### 1.4 예약 상태 조정과 취소
- **상태 변경**: 관리자는 예약 상태 글자를 직접 바꿀 수 있으며, 값에 대한 제한이 없다.
- **본인 확인 절차 없음**: 취소나 수정 시 확인 코드를 확인하지 않는다.
- **환불 계산**: 당일 취소·사전 취소 구분과 환불 금액 계산 기능이 준비되지 않았다.

## 2. 캠핑 구역 관리

### 2.1 구역 정보 유지
- **핵심 정보**: 각 구역은 고유 번호, 간단한 설명, 최대 이용 인원으로 관리된다.
- **고유 조건**: 번호가 겹치면 저장되지 않으므로, 새 구역 등록 전 중복 여부를 확인해야 한다.
- **구역 유형**: 번호 접두어로 대형·소형을 구분하는 자동 로직은 없다.

### 2.2 구역 등록과 수정
- **등록 흐름**: 담당자는 구역 번호, 설명, 최대 인원을 입력해 새 구역을 추가할 수 있다.
- **부분 수정**: 필요 항목만 골라 수정할 수 있으며, 잘못된 숫자 값은 무시되고 기존 값이 유지된다.
- **화면 연동**: 관리자 화면에서도 동일한 절차를 지원해 현장에서 바로 등록·수정할 수 있다.

### 2.3 가용성 판단
- **빈자리 계산 없음**: 특정 날짜나 기간에 비어 있는 구역을 자동으로 알려주지 않는다.
- **연박 가용성**: 연속 숙박이 가능한지 확인하려면 예약 목록을 직접 비교해야 한다.
- **필터링 미지원**: 구역 크기나 편의시설 조건으로 걸러 보는 기능이 없다.

## 3. 상품, 대여, 재고 관리

### 3.1 상품 재고
- **상품 기본 정보**: 상품 이름, 재고 수량, 판매 단가, 판매/대여 구분을 저장한다.
- **재고 보호**: 재고보다 많은 수량을 판매하거나 대여하려 하면 작업이 중단된다.
- **유형 변경**: 판매 상품을 대여 상품으로 바꾸는 변경 흐름이 존재하며, 자동 테스트로 검증되고 있다.

### 3.2 대여 처리
- **대여 등록**: 담당자는 대여 품목, 수량, 연결된 예약(선택)을 입력해 대여 기록을 남길 수 있다.
- **예약 연동 선택**: 예약과 연결하지 않고 현장 대여만 기록하는 것도 가능하다.
- **반납 절차**: 반납 처리 시 재고가 즉시 되돌아가며, 이미 반납된 기록을 다시 반납하려 하면 거절된다.

### 3.3 판매 처리와 영수증
- **판매 등록**: 여러 품목을 한 번에 판매 기록으로 남길 수 있으며, 각 품목별 재고 차감과 매출 기록이 동시에 적용된다.
- **판매 내역 조회**: 최근 판매 내역을 최신순으로 확인할 수 있다.

## 4. 매출 보고와 운영 지원

### 4.1 매출 요약 보고
- **일별 보고**: 숙박, 판매, 대여 매출을 하루 단위로 집계해 보여준다.
- **기간 보고**: 시작일과 마감일을 지정해 같은 기준으로 합산 결과를 확인할 수 있다.
- **세부 항목**: 지정 기간 동안 발생한 예약·판매·대여를 각각 시간순으로 나열해 추적할 수 있다.

### 4.2 숙박 단가 정책
- **표준 단가**: 예약 매출을 계산할 때 1박당 5만 원을 기준으로 사용한다.
- **정책 변경 시 고려**: 단가가 코드에 고정되어 있어, 요금 정책이 바뀌면 개발 수정이 필요하다.

### 4.3 데이터 초기화와 로그
- **초기 데이터**: 시스템이 켜질 때마다 준비된 예약, 상품, 대여 기록이 자동으로 채워져 시연과 테스트에 활용된다.
- **로그 수준**: 주요 업무 흐름과 데이터베이스 질의가 자세히 기록돼 문제 원인을 추적하기 쉽다.

## 5. 보안과 접근 제어

### 5.1 관리자 로그인
- **인증 방식**: 관리자 이름과 비밀번호가 맞아야만 시스템에 들어갈 수 있다.
- **접근 제한**: 로그인에 성공해야만 예약, 상품, 매출 관리 화면을 사용할 수 있다.

### 5.2 세션 유지
- **인증 유지 시간**: 로그인 후 일정 시간이 지나면 다시 로그인해야 한다.
- **로그아웃 처리**: 로그아웃 시 인증 정보가 즉시 사라져, 다른 사람이 같은 기기를 사용해도 접근할 수 없다.

## 6. 비즈니스 규칙 요약

1. 관리 화면은 인증된 관리자만 사용할 수 있다.
2. 예약은 이름, 시작일, 마감일, 캠핑 구역이 모두 채워져야 한다.
3. 새 예약은 자동으로 확정 상태로 저장된다.
4. 전화번호와 확인 코드는 운영자가 필요에 따라 직접 입력한다.
5. 동일한 캠핑 구역 번호는 중복으로 등록할 수 없다.
6. 재고보다 많은 수량으로 판매나 대여를 진행할 수 없다.
7. 숙박 매출은 1박당 5만 원 기준으로 계산된다.
8. 예약 일정 검증과 연박 검증은 사람이 직접 확인해야 한다.

## 상위 위험 기능 메모

- **예약 생성 및 검증**: 자동 검증이 부족해 잘못된 일정이나 중복 예약이 그대로 반영될 위험이 크다.
- **가용성 판단**: 빈자리 계산이 수동으로 이뤄져, 성수기에는 안내 오류가 매출 손실로 이어질 수 있다.
- **매출 보고**: 숙박 단가가 고정되어 있어 정책 변경 시 지표가 왜곡될 가능성이 있다.
