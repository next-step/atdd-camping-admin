# API 리스크 평가 및 우선순위 도출

## 1. 평가 기준

각 기능을 다음 세 가지 기준으로 평가하여 점수화(1~3점)하고, 이를 합산하여 리스크 총점을 산출했습니다.

| 기준 | 설명 | 3점 (높음) | 2점 (보통) | 1점 (낮음) |
| :--- | :--- | :--- | :--- | :--- |
| **비즈니스 임팩트** | 기능 장애 시 비즈니스에 미치는 영향도 | 매출/신뢰도 직결, 핵심 프로세스 중단 | 운영 불편, 일부 기능 제한 | 단순 조회, 대안 존재 |
| **변경 빈도** | 요구사항 변경이나 로직 수정 빈도 | 자주 변경됨, 정책이 유동적임 | 가끔 변경됨 | 거의 변경되지 않음 (초기 설정 등) |
| **버그 가능성** | 코드 복잡도, 예외 처리, 동시성 이슈 등 | 복잡한 로직, 검증 미흡, 동시성 이슈 존재 | 보통의 복잡도, 일반적인 CRUD | 단순 로직, 방어 코드 충분 |

## 2. API별 상세 평가

### 1) 대여 생성 (`POST /admin/rentals`)
*   **비즈니스 임팩트 (3점)**: 재고 차감 및 매출과 직결됩니다. 오류 발생 시 재고 불일치로 인한 현장 컴플레인, 매출 누락이 발생합니다.
*   **변경 빈도 (2점)**: 대여 정책, 가격 로직, 재고 관리 방식 등은 운영 상황에 따라 조정될 여지가 있습니다.
*   **버그 가능성 (3점)**: **가장 위험함.** 재고 차감 시 동시성 이슈(Race Condition) 발생 가능성이 높습니다. 상품 타입 검증(`RENTAL`), 예약 ID 유무에 따른 분기 처리 등 로직이 복잡합니다.
*   **총점: 8점**

### 2) 예약 상태 변경 (`PATCH /admin/reservations/{id}/status`)
*   **비즈니스 임팩트 (3점)**: 예약은 캠핑장 운영의 핵심입니다. 잘못된 취소나 확정은 고객의 신뢰를 치명적으로 손상시킵니다.
*   **변경 빈도 (2점)**: 상태 전이 규칙(예: `WAITING` -> `PAID` -> `CONFIRMED`)은 비즈니스 흐름에 따라 변경될 수 있습니다.
*   **버그 가능성 (3점)**: **데이터 무결성 위험.** 입력값(`status`)에 대한 Enum 검증이 없고 단순 문자열로 저장하므로, 오타나 유효하지 않은 값이 DB에 저장될 위험이 매우 큽니다.
*   **총점: 8점**

### 3) 상품 생성 (`POST /admin/products`)
*   **비즈니스 임팩트 (2점)**: 상품 등록이 안 되면 판매를 할 수 없으나, 대여나 예약만큼 실시간 트랜잭션 빈도가 높지는 않습니다.
*   **변경 빈도 (2점)**: 상품 속성(카테고리, 가격 정책 등) 추가 등의 유지보수 소요가 있을 수 있습니다.
*   **버그 가능성 (2점)**: **방어 로직의 역효과.** 파싱 실패 시 에러를 내지 않고 기본값(`0`, `SALE`)으로 덮어쓰는 로직이 있어, 클라이언트가 잘못된 데이터를 보내도 '성공'한 것처럼 보이는 논리적 버그(Silent Failure) 가능성이 있습니다.
*   **총점: 6점**

### 4) 캠프사이트 생성 (`POST /admin/campsites`)
*   **비즈니스 임팩트 (3점)**: 사이트 정보는 예약의 기반 데이터이므로 중요도가 높습니다. 잘못 설정되면 중복 예약 등의 시발점이 됩니다.
*   **변경 빈도 (1점)**: 캠핑장의 물리적 사이트 구성은 거의 변하지 않는 정적 데이터입니다.
*   **버그 가능성 (1점)**: 로직이 매우 단순한 CRUD이며, 실패 시 로직이 복잡하게 꼬일 여지가 적습니다.
*   **총점: 5점**

### 5) 예약 목록 조회 (`GET /admin/reservations`)
*   **비즈니스 임팩트 (2점)**: 현황 파악에 필수적이나, 조회가 잠시 안 된다고 해서 데이터가 오염되거나 금전적 손실이 발생하지는 않습니다.
*   **변경 빈도 (1점)**: 조회 요건은 크게 변하지 않으며, 필터링 추가 정도의 변경만 예상됩니다.
*   **버그 가능성 (1점)**: 단순 조회 로직이며 Null 처리가 되어 있어 런타임 에러 가능성이 낮습니다. (단, 데이터 증가 시 성능 이슈는 별개)
*   **총점: 4점**

## 3. 종합 리스크 우선순위 (Risk Ranking)

리스크 총점이 높은 순서대로 우선순위를 매겼습니다. **테스트 코드 작성 및 리팩토링 시 상위 순위 기능에 집중해야 합니다.**

| 순위 | API 기능 | 총점 | 비고 |
| :--- | :--- | :--- | :--- |
| **1** | **대여 생성** | **8** | **[최우선 관리 필요]** 동시성 제어 및 재고 정합성 테스트 필수 |
| **2** | **예약 상태 변경** | **8** | **[데이터 무결성 주의]** 상태값 유효성 검증(Validation) 로직 추가 시급 |
| **3** | 상품 생성 | 6 | 잘못된 입력에 대한 예외 처리 강화 권장 |
| **4** | 캠프사이트 생성 | 5 | 초기 데이터 셋업 시 정확성 검증 필요 |
| **5** | 예약 목록 조회 | 4 | 안정적임. 성능 테스트 정도 고려 |
